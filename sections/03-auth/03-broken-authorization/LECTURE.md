## 学ぶこと

「ログインさえしていれば誰でもアクセスして良い」という実装は、**認証 (authentication) は通っているが認可 (authorization) が抜けている** 状態です。OWASP Top 10 でも **Broken Access Control** が連年首位という、現場で最も多いパターンの 1 つです。

この章で身につけたい観点は次の通りです。

- **認証** と **認可** は別物。認証はあなたが誰かを確かめる、認可はそれをして良い権限があるかを確かめる
- 同じロールの別ユーザーのデータを覗ける **水平権限の不備（IDOR）** と、一般ユーザーが管理者機能に到達できる **垂直権限の不備** がある
- 画面に出さないだけでは不十分。**DB から取らない** のが最小権限の原則

## 説明

### TODO 1: 他人の情報を抜く

`tanaka@example.com / password` でログインしたあと、アドレスバーに以下を順に入力してください。

- `/users/1`（管理者 admin）
- `/users/3`
- `/users/15`

それぞれのページに、本来 **その人にしか見せるべきでない情報** が表示されています。

| 表示されている項目 | 中身 |
|---|---|
| 内部 ID | DB の主キー |
| メールアドレス | ログインに使う識別子 |
| ロール | `admin` / `user` |
| 登録日 | アカウント作成日時 |
| パスワードハッシュ（先頭16文字） | scrypt の `salt:hash` 形式 |

パスワードハッシュは「ハッシュだから安全」ではありません。**salt が分かる** ことや **アルゴリズムが推定できる** ことは、レインボーテーブル攻撃や辞書攻撃の手がかりになります。たとえ一部であっても、外には出すべきではない情報です。

### TODO 2: なぜ漏れるか

該当箇所は [app/routes/users.js](./app/routes/users.js) です。

```js
app.use('*', requireAuth());  // 認証はしている
const profile = db.prepare(
  'SELECT id, name, email, role, password, created_at FROM users WHERE id = ?'
).get(id);
// ↑ password まで取る必要は無い
// ↓ session.userId と id の照合が無いまま全情報を表示
```

ポイントは 2 つあります。

- **認可チェックがない**：`requireAuth()` は「ログインしているか」しか見ません。`session.userId === Number(id)` のような **「リソースの所有者かどうか」** の照合がないため、ログインユーザーは誰でも他人のページを開けてしまいます。これが **水平権限の不備（IDOR: Insecure Direct Object Reference）** です。
- **取得カラムが過剰**：`SELECT` で `password` まで含めて取得しています。HTML に出さない設計にしたとしても、**DB から取った時点で漏れの素** です。エラー応答にオブジェクトを丸ごと返したり、ログ出力したり、JSON API に転用したり、後から事故が起きるルートが残ります。

### TODO 3: 修正の3点

3 つの観点を組み合わせて直します。

- **(a) 取得カラムを必要最小限にする**：`password` は最初から SELECT しない
- **(b) 見せ方を切り分ける**：自分のページならメールも出す。他人のページなら公開情報（名前・ロール・記事一覧）だけ
- **(c) 管理者は別**：`currentUser.role === 'admin'` なら全部見せる

修正例:

```js
const isSelf = currentUser.id === Number(id);
const isAdmin = currentUser.role === 'admin';

const cols = (isSelf || isAdmin)
  ? 'id, name, email, role, created_at'
  : 'id, name, role, created_at';
const profile = db.prepare(`SELECT ${cols} FROM users WHERE id = ?`).get(id);
// ※ ホワイトリストで cols を切り替えているので SQL インジェクションにはならない
```

表示側も `isSelf || isAdmin` でメール行を出し分け、`password`（ハッシュ）の表示は完全に消します。

修正後、再び `tanaka` で `/users/1` を開き、メールアドレスやパスワードハッシュが **見えなくなっている** ことを確認してください。一方 `tanaka` 自身（`/users/<tanaka の id>`）や `admin` でログインしたときは詳細が見える、という挙動になっていれば成功です。

## 補足: 認証と認可の言い分け

- **認証 (authentication / authn)**：あなたが誰かを確かめる。ログイン処理がこれにあたる
- **認可 (authorization / authz)**：あなたがそれをする権限があるかを確かめる。リソースの所有者かどうか、必要なロールを持っているかどうか

「認証通ってるんだから OK」と認可を省くのが本レクチャーの罠です。OWASP Top 10 で **Broken Access Control** が連年首位なのは、ログインまでで満足する実装が今も多いからです。

## 補足: 水平権限と垂直権限

- **水平権限の不備 (IDOR)**：同じ「一般ユーザー」同士で他人のデータを見るパターン。`/users/123` の id を変えるだけで他人の情報が出るのが典型。本章はこちら
- **垂直権限の不備**：一般ユーザーが管理者画面・管理者 API にアクセスできるパターン（前のレクチャーで扱った）

どちらも対策の基本は同じで、**サーバー側でリソース ID と現在ユーザーの紐付き／ロールを毎回確認する** ことです。クライアントから来た id を信用しない、UI でボタンを隠したから安全とは考えない、というのが鉄則です。

## 補足: 「画面に出さなければいい」ではなく SQL で取らない

画面の HTML から `password` の表示行だけ消す、という直し方では足りません。

- DB から取得したオブジェクトが、JSON ログやエラー画面にそのまま乗ることがある
- 後から「API 用に同じハンドラを使い回そう」という改修が入った瞬間に漏れる
- キャッシュやデバッグ出力に紛れ込む

**DB から取得する段階で必要な列だけにする** のが最小権限の原則です。`SELECT *` を避け、列を明示するだけでも事故率は大きく下がります。
