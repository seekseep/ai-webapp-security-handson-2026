## 学ぶこと

管理画面のように **誰でも見られては困るページ** は、サーバー側で「ログインしているか」「管理者ロールを持っているか」を必ずチェックし、条件を満たさないリクエストは別の URL にリダイレクトする必要があります。フロントから `/admin` リンクを消したり、URL を推測しづらくするだけでは隠ぺいにしかなりません。

この章で身につけたい観点は次の3つです。

- ログイン状態・ロールを `c.get('user')` から取り出してチェックする書き方
- 「ハンドラの先頭で未認証ならリダイレクトして処理を打ち切る」という基本パターン
- 同じチェックを **エンドポイントの数だけ書き写す** 冗長さを体で覚えること（次節で「これを1か所にまとめるのがミドルウェア」という気付きに繋げます）

## 説明

### TODO 1: 未ログインで `/admin` にアクセスする

ログアウトした状態（あるいはシークレットウィンドウ）でブラウザを開き直し、http://localhost:3000/admin に直接アクセスしてください。

- ユーザー一覧テーブルが表示される（メールアドレス・ロール・登録日まで丸見え）
- 一般ユーザー（`tanaka@example.com`）でログインすると、ヘッダーに `/admin` リンクは出ないものの URL 直打ちで届く

つまり `/admin` 配下のすべてのエンドポイントが、ログインの有無やロールに関係なく叩ける状態です。

### TODO 2: なぜ素通しになっているのかを説明する

該当箇所は [app/routes/admin.js](./app/routes/admin.js) です。冒頭から各ハンドラの中まで眺めると、**ログイン状態やロールをチェックしている箇所がどこにも無い** ことが分かります。

```js
// 管理画面トップ（ユーザー一覧）
app.get('/', (c) => {
  const user = c.get('user');
  const users = db.prepare('SELECT id, name, email, role, created_at FROM users ORDER BY id').all();
  // ...そのまま一覧を返してしまう
});

// ユーザー削除
app.post('/users/:id/delete', (c) => {
  const user = c.get('user');
  const targetId = Number(c.req.param('id'));
  // ...user が null でも処理が進んでしまう
});
```

`c.get('user')` には、別途設定された `currentUser()` の働きでログインユーザー（または `null`）が入っています。`null` なら未ログイン、`role` が `'admin'` でなければ一般ユーザー、ということが分かるので、これを使って弾くコードを各ハンドラの先頭に書き足します。

### TODO 3: 各ハンドラの先頭に直接書く

[app/routes/admin.js](./app/routes/admin.js) の **2つのハンドラそれぞれの先頭** に、次のチェックを書き写してください。「未ログイン、または管理者ロールでない」場合はトップ（`/`）にリダイレクトして処理を打ち切る、という内容です。

```js
// 管理画面トップ（ユーザー一覧）
app.get('/', (c) => {
  const user = c.get('user');
  if (!user || user.role !== 'admin') {
    return c.redirect('/');
  }

  const users = db.prepare('SELECT id, name, email, role, created_at FROM users ORDER BY id').all();
  // ...以下はそのまま
});

// ユーザー削除
app.post('/users/:id/delete', (c) => {
  const user = c.get('user');
  if (!user || user.role !== 'admin') {
    return c.redirect('/');
  }

  const targetId = Number(c.req.param('id'));
  // ...以下はそのまま
});
```

ポイントは **同じ4行を2つのハンドラに書き写している** ことです。`return` を忘れる、片方のハンドラに書き忘れる、`role !== 'admin'` を `role === 'admin'` と打ち間違える、といったミスがそのまま「全公開」につながるので、書きながら冗長さと危うさの両方を意識してください。

修正後、もう一度未ログインで http://localhost:3000/admin を開くと `/` にリダイレクトされ、ユーザー一覧が見えなくなります。`tanaka@example.com`（一般ユーザー）でログインしている状態でも同じく弾かれ、`admin@example.com` でログインしているときだけ管理画面が表示される、という状態になっていれば成功です。

## 補足: この冗長さを1か所にまとめるのが「ミドルウェア」

今回は `admin.js` のハンドラが2つだけだったので「同じ4行を2回書き写す」程度で済みました。しかし実運用では、管理画面に10も20もエンドポイントが並ぶことは普通にあります。そのすべての先頭に同じチェックを書き写すのは現実的ではありませんし、**書き忘れた1か所がそのまま情報漏えいや権限昇格に直結** します。

この「ルートの前段で必ず通る共通処理」を1か所にまとめる仕組みが **ミドルウェア** です。Hono であれば、ルーターの先頭で次のように1行書くだけで、その配下のすべてのルートに同じガードが当たります。

```js
app.use('*', requireAdmin());
```

ミドルウェアを使った「きれいな書き方」は [01-environment/01-environment](../../01-environment/01-environment/app/routes/admin.js) の `admin.js` を見てください。今回の章で書き写した4行が `requireAdmin()` という1つの関数にまとまり、ルーターに1度当てるだけで全エンドポイントに効いている様子が確認できます。

ミドルウェアそのものをどう作るか・どこに当てるかは、この後の章で扱っていきます。ここでは「共通処理を全ハンドラに書き写すのはツラいし危ない」という肌感覚を持っておけば十分です。

## 補足: 「URL を知らないから安全」は通用しない

`/admin` リンクをヘッダーから外したり、推測しづらい URL に変えたりするだけでは隠ぺいにしかなりません。攻撃者は URL 推測やクローリング、JavaScript の解析、過去のリーク情報などから内部パスに容易に到達します。

いわゆる **Security through obscurity（隠ぺいによるセキュリティ）** に頼らず、サーバー側で必ず認可チェックを行う、という原則を覚えておいてください。フロントから導線を消すのは UX の整理であって、セキュリティ対策ではありません。
