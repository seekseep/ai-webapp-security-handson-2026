## 学ぶこと

「ログインさえしていれば誰でもアクセスして良い」という実装は、**認証 (authentication) は通っているが認可 (authorization) が抜けている** 状態です。[OWASP](https://owasp.org/www-chapter-japan/) Top 10 でも **Broken Access Control** が連年首位という、現場で最も多いパターンの 1 つです。

この章で身につけたい観点は次の通りです。

- **認証** と **認可** は別物。認証はあなたが誰かを確かめる、認可はそれをして良い権限があるかを確かめる
- 同じロールの別ユーザーのデータを覗ける **水平権限の不備（IDOR）** と、一般ユーザーが管理者機能に到達できる **垂直権限の不備** がある
- 「ログインさえしていれば OK」では不十分で、リクエストごとに **そのリソースに対する権限** を確認する必要がある

## 説明

### TODO 1: 他人の情報を抜く

`tanaka@example.com / password` でログインしたあと、アドレスバーに以下を順に入力してください。

- `/users/1`（管理者 admin）
- `/users/3`
- `/users/15`

それぞれのページに、本来 **その人にしか見せるべきでない情報** が表示されています。

| 表示されている項目 | 中身 |
|---|---|
| 内部 ID | DB の主キー |
| メールアドレス | ログインに使う識別子 |
| ロール | `admin` / `user` |
| 登録日 | アカウント作成日時 |

特にメールアドレスは **ログインに使う識別子** であり、フィッシングや総当たり攻撃の入り口にもなります。他人のページで見えていい情報ではありません。

### TODO 2: なぜ漏れるか

該当箇所は [app/routes/users.js](./app/routes/users.js) です。

```js
app.use('*', requireAuth());  // 認証はしている
const profile = db.prepare(
  'SELECT id, name, email, role, created_at FROM users WHERE id = ?'
).get(id);
// ↓ currentUser.id と id の照合が無いまま全情報を表示
```

ポイントは **認可チェックがない** ことです。`requireAuth()` は「ログインしているか」しか見ません。`currentUser.id === Number(id)` のような **「リソースの所有者かどうか」** の照合がないため、ログインユーザーは誰でも他人のページを開けてしまいます。これが **水平権限の不備（IDOR: Insecure Direct Object Reference）** です。

### TODO 3: 修正

「閲覧者と対象者の関係」と「閲覧者のロール」を見て、メールアドレスの表示を出し分けます。整理すると次の表になります。

| 閲覧者のロール | 対象ユーザーとの関係 | 認可の根拠 | メール表示 |
|---|---|---|---|
| 一般ユーザー | 自分自身 (`isSelf`) | リソースの所有者なので許可 | 表示する |
| 一般ユーザー | 他人 | 所有者でもなく特権もない | 隠す |
| 管理者 (`admin`) | 誰でも | 管理者ロールで許可 | 表示する |

つまり許可の条件は **`isSelf || isAdmin`** の 1 つにまとまります。

修正例:

```js
const isSelf = currentUser.id === Number(id);
const isAdmin = currentUser.role === 'admin';
```

そして HTML 側でメール行を出し分けます。

```js
${(isSelf || isAdmin) ? html`<p>メール: ${profile.email}</p>` : ''}
```

修正後、再び `tanaka` で `/users/1` を開き、メールアドレスが **見えなくなっている** ことを確認してください。一方 `tanaka` 自身（`/users/<tanaka の id>`）や `admin` でログインしたときはメールが見える、という挙動になっていれば成功です。

## 補足: 認証と認可の言い分け

- **認証 (authentication / authn)**：あなたが誰かを確かめる。ログイン処理がこれにあたる
- **認可 (authorization / authz)**：あなたがそれをする権限があるかを確かめる。リソースの所有者かどうか、必要なロールを持っているかどうか

「認証通ってるんだから OK」と認可を省くのが本レクチャーの罠です。[OWASP](https://owasp.org/www-chapter-japan/) Top 10 で **Broken Access Control** が連年首位なのは、ログインまでで満足する実装が今も多いからです。
